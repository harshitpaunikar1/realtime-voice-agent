Real-Time Voice Agent

A room conversation is not judged on average intelligence. It is judged on the gap between a human mouth closing and a synthetic voice beginning. This document treats the voice agent as a measured pipeline, not a character, and follows latency from microphone ingress to interruption recovery.

t+000 ms

// Delay Is the Product

In a text interface, a pause can hide behind a spinner. In a room, it sits in the air. The user has finished speaking, the speaker cone is silent, and the system has a fraction of a second to prove that it was listening rather than buffering. That is why the correct opening frame for this project is not model selection or prompt design. It is conversational delay.

The scope matters. One user, one AI agent, one room. This removes multi-party arbitration, cross-room fan-out, and diarization. It does not remove timing pressure. A single-user room is still a real-time system because turn-taking is embodied. A pause around 200 ms feels adjacent to natural overlap. A pause near 500 ms becomes noticeable. A pause pushing toward a second starts to feel like a network hop, not a local exchange. At that point users change strategy: they repeat themselves, they simplify questions, or they talk over the agent. Those behaviors are not user error. They are latency symptoms.

The architecture named in the README is straightforward on paper. LiveKit moves audio in and out of the room. faster-whisper converts speech to text. Gemini Flash produces the response. Piper turns the response back into speech. The important part is what the paper diagram hides: every box adds framing, buffering, queueing, and handoff cost before any model quality is even relevant. The system only feels fast when the sum of those small waits is kept inside a budget and when adjacent stages overlap instead of serialize.

This analysis therefore treats the pipeline as a set of measured segments. Audio ingress has a cost. Endpointing has a cost. Transcription has a cost. Response planning has a cost. Speech synthesis has a cost. Playback has a cost. Interruption recovery has its own budget and must remain independent from the clean-turn path. There is no magic real-time property conferred by the word streaming. There is only cumulative latency, stage by stage, under load and in failure.

Target operating shape: first audible response under 900 ms on a clean turn, full clause emission while the remainder of the answer is still being generated, and interruption stop time below 150 ms so the user can retake the floor without friction.

t+144 ms

// Audio Ingress And Room Edge

The pipeline starts before transcription. It starts when acoustic energy becomes transportable frames. In a LiveKit room that means browser capture cadence, encoding, packet transit, server handling, and room event timing all happen before language enters the picture. When people say a voice agent feels sluggish, they often blame the model because that is the visible brand in the stack. The first avoidable waits usually live earlier.

The clean-turn budget above allocates 32 ms to local capture and 44 ms to transport. Those numbers are not theoretical minimums; they are a realistic compact window for a one-room path without cross-region detours. Capture cost appears small because the browser is already recording continuously. The actual penalty is frame accumulation and handoff. Transport cost also appears small because LiveKit is solving a hard media problem efficiently, but it is still subject to jitter, queueing, and room state changes. If that transport path stretches from 44 ms toward 90 ms, the rest of the system inherits the delay without any model having done additional work.

Endpointing is the first major policy decision. The project centers one user and one agent, so the system does not need to infer which speaker owns the turn. It only needs to decide when enough silence or enough confidence exists to commit the utterance downstream. That is where 68 ms is spent in the representative path: not on compute-heavy intelligence, but on deciding that the user has probably yielded the floor. Set the threshold too aggressively and you clip trailing words. Set it too conservatively and you manufacture dead air. Either mistake is immediately audible.

The correct implementation posture is to log room events and media-stage timestamps independently. A join event, track subscription event, voice activity transition, and utterance commit should not be conflated into one generic "received audio" marker. The point of the logging surface mentioned in the README is not archival completeness. It is causal clarity. When a turn slips from 866 ms to 1280 ms, the operator needs to know whether the extra 414 ms came from capture jitter, room transport, endpoint hesitation, STT backlog, or synthesis queueing.

from time import perf_counter_ns

class TurnTrace:
    def __init__(self, turn_id: str) -> None:
        self.turn_id = turn_id
        self.t0 = perf_counter_ns()
        self.events = []

    def mark(self, name: str, **fields) -> None:
        now = perf_counter_ns()
        self.events.append({
            "turn_id": self.turn_id,
            "event": name,
            "ms": round((now - self.t0) / 1_000_000, 2),
            **fields,
        })

trace = TurnTrace(turn_id)
trace.mark("room.audio_frame")
trace.mark("vad.speech_end")
trace.mark("turn.commit")

That snippet is intentionally plain. The system does not need ornate telemetry to become diagnosable. It needs one clock origin per turn and consistent stage names. Once the ingress side is measurable, the later model stages can be judged fairly instead of absorbing blame for transport uncertainty they did not create.

t+322 ms

// faster-whisper As A Commit Surface

Speech-to-text is often drawn as a single rectangle labeled STT. That is too coarse for latency work. The stage actually contains chunk collection, decoding, partial hypothesis formation, and finalization logic. In this system faster-whisper is the first language-bearing stage, but it should not be treated as the first moment where timing discipline matters. It inherits all upstream uncertainty and then decides how much of that uncertainty it passes along.

The representative budget gives faster-whisper 178 ms after utterance commit to produce a usable transcription. That is a narrow budget, but it is sufficient when the system stays aligned with the product scope. One user means no diarization. One room means no cross-participant arbitration. A practical voice agent does not need maximal transcription ceremony. It needs a stable clause quickly enough to unblock response planning. That usually means smaller audio slices, fixed language when known, and a bias toward incremental emission rather than waiting for perfect sentence closure.

The main mistake at this stage is confusing transcript prettiness with transcript utility. A punctuation-clean final string is valuable for logs. It is not necessary for first-response timing. The backend should separate the "safe to plan against" condition from the "final archival text is ready" condition. If the system waits for transcript cleanup, number normalization, and post-formatting before the LLM sees anything, STT turns from an enabling stage into a queue. In real-time voice work, small formatting wins are not worth 150 ms of dead air.

There is also an asymmetry between clean turns and interruption recovery. On a clean turn, the STT stage can wait briefly for confidence. On an interrupted turn, it must reacquire faster because the user is usually correcting the agent or clarifying a failed answer. The system therefore benefits from preserving partial decoder state and from carrying the interruption marker into STT logs so false restarts are visible. Without that annotation, operators see longer transcribe times without knowing whether the model was slow or the turn was restarted mid-stream.

Operational rule: use partial transcripts to prepare, but only hand Gemini Flash a segment that has crossed an explicit commit condition. Fast and wrong is not conversational; it is merely early.

What this buys the pipeline is not raw STT speed in isolation. It buys a predictable handoff. Once a committed text segment exists around 322 ms cumulative, the next stage can begin with a bounded uncertainty envelope. That is the difference between a voice stack that appears to improvise and one that behaves like a designed system.

t+569 ms

// Gemini Flash Under A Latency Budget

Once text is committed, the system enters the reasoning stage. Here the temptation is to discuss prompt craft in abstract terms. For a latency-first analysis, the more important question is what kind of answer structure Gemini Flash can return quickly enough to feed speech synthesis without audible hesitation. A voice agent does not need a finished essay before it speaks. It needs a first clause that is semantically safe, locally coherent, and unlikely to be invalidated by the remainder of generation.

The budget assigns 247 ms to this stage, carrying cumulative time from 322 ms to 569 ms. That number implies aggressive scope control. Prompts must be short. Retrieved context, if any, must be bounded. Output length must be capped or at least steered toward concise spoken language. The point is not to make the model terse for aesthetic reasons. The point is that every extra token can either delay first-clause stability or expand the TTS workload that follows. In a room interaction, verbosity has a direct acoustic cost.

There is another constraint unique to speech: the first text chunk must sound like the beginning of a sentence a human would say aloud. Many text-first systems can stream jagged fragments because the eye is tolerant. The ear is less forgiving. Gemini Flash therefore should be used here less as a maximal reasoner and more as a low-latency planner that emits clause-shaped units. If the first chunk is "Well, based on" and then stalls, the user hears uncertainty. If the first chunk is a compact answer core, Piper can begin producing sound while later reasoning continues.

That makes overlap the central optimization. The ideal path is not STT complete, then full reasoning complete, then full synthesis complete. The ideal path is STT commit, then first clause generation, then immediate handoff to TTS while the remainder of the response is still being produced. This is the only honest way to keep cumulative latency low without cutting answer quality to the floor. A system that insists on full serial completion will always pay the full width of every stage in the ribbon.

Latency budgets also clarify failure policy. If Gemini Flash has not produced a synthesis-safe clause within its allocation window, the system needs a deterministic fallback rather than a silent wait. That fallback might be a shorter prompt, a capped retry, or a degraded answer template. The anti-magic principle matters here: the user should encounter a bounded, simpler answer rather than an apparently arbitrary pause whose cause is invisible.

t+866 ms

// Piper, Queueing, And Audible Start

Text-to-speech is where the system finally becomes accountable to the room again. Until this point, the pipeline has been accumulating invisible work. Piper turns that work into an audible commitment. If the first waveform is late, every prior stage becomes visible at once. If the first waveform is early but poorly buffered, the system sounds broken instead of slow. Either way the user experiences the sum, not the internal excuse.

The representative path gives Piper 161 ms to synthesize the first usable chunk, 38 ms to packetize and queue it, and 98 ms for playback buffering before the speaker emits the first phoneme at 866 ms cumulative. These numbers show why TTS should be treated as a streaming stage, not a file export. Waiting for the full response text and then asking Piper for a complete audio asset is structurally incompatible with conversational timing. The correct unit is the first clause. Everything after that should overlap: later text generation, later synthesis, and incremental playback.

Playback itself deserves explicit ownership. The room layer is not finished once bytes leave Piper. Audio still needs framing, enqueueing, and a buffer large enough to prevent stutter without reintroducing the pause the earlier stages worked to remove. In practice this means the system must decide how much audio to hold before starting playback. Too little and jitter or synthesis variance causes audible under-runs. Too much and the user waits through a silent reserve that provides no conversational value. The 98 ms playback slice in the ribbon is not spare time. It is a negotiated compromise between smoothness and responsiveness.

This stage is also where answer length becomes financially real. An overlong text reply is not just an LLM style issue. It increases synthesis CPU time, expands outbound buffering, and stretches the interval during which interruption must remain clean. Spoken systems should therefore bias toward compact first turns and progressive elaboration. A short answer that can extend on request is acoustically cheaper and easier to interrupt than a single dense monologue.

That is why the pipeline should be optimized against the time to audible speech, not against isolated model timings that look impressive in a dashboard but do not map to the human exchange.

barge-in path

// Turn-Taking And Interruption Recovery

A voice agent that cannot be interrupted is not participating in conversation. It is broadcasting. The room constraint sharpens this further because there is only one user to satisfy. If that user starts speaking while the agent is mid-sentence, the system does not need democratic fairness. It needs to yield quickly and cleanly. Interruption handling is therefore not an accessory feature added after the main pipeline works. It is part of the main pipeline.

The state machine below makes the expected transitions explicit. Listen leads to transcribe once speech is detected and a turn is committed. Transcribe leads to think when the text segment is stable enough for response planning. Think leads to speak when a synthesis-safe clause exists. While speaking, a new user voice event should not be treated as ambient noise by default. It should be tested as a potential barge-in. If confirmed, the agent enters interrupt, cancels synthesis and playback with bounded drain behavior, then moves to recover so the user can retake the floor without carrying stale audio forward.

The operational challenge is not merely canceling work. It is canceling work at the right boundaries. If playback drains too much queued audio, the user still hears the agent talking after they have started. If playback stops too abruptly without packet discipline, the room clicks or tears. If reasoning continues after interruption, the backend may synthesize a stale answer for a turn the user has already replaced. Recovery therefore needs explicit ownership of playback, TTS, and reasoning cancellation, plus a clear rule for what state survives the cut.

async def handle_barge_in(turn_id: str, frame_energy_db: float) -> None:
    if state.mode != "speak" or frame_energy_db < BARGE_IN_DB:
        return

    trace.mark("barge_in.detected", turn_id=turn_id)
    state.mode = "interrupt"

    await playback.stop_after_current_packet()
    await tts.abort(turn_id)
    await llm.abort(turn_id)

    trace.mark("barge_in.cleared")
    stt.begin_recovery_window()
    state.mode = "recover"

The recovery window is a systems detail worth naming. After interruption, the pipeline should favor rapid reacquisition over deep cleanup. The user is usually correcting course. That means fast speech detection, compact transcription chunks, and strong trace linkage between the interrupted turn and the replacement turn. Without that linkage, optimization efforts cannot distinguish "slow normal reply" from "normal interruption handling." Those are different phenomena and should remain different in the logs.

// Tracing, Diagnosis, Optimization

The README notes that room and pipeline events are logged for debugging. That is the correct closing point for the architecture because a latency-first voice stack is only as good as its trace discipline. The system should emit events that preserve causal order across the entire path: room join, audio frame ingress, speech start, speech end, utterance commit, transcript partial, transcript stable, reasoning start, first clause ready, synthesis start, first audio chunk, playback start, interruption detected, playback stop, recovery start. If those events share a turn identifier and a single origin clock, most performance disputes become answerable rather than rhetorical.

A useful trace is not only granular; it is budget-aware. Operators should be able to compare actual stage times against expected envelopes at a glance. The clean-turn example in this page gives one such envelope: 144 ms to commit, 322 ms to stable text, 569 ms to response clause, 866 ms to audible speech. If a run misses the envelope, the trace should make the miss local. "The agent was slow" is not a diagnosis. "Commit drifted by 90 ms because endpoint silence threshold was too conservative" is a diagnosis. So is "reasoning exceeded budget because reply length doubled after prompt expansion." So is "playback start slipped because outbound audio queue waited for a full sentence instead of a clause."

Optimization then becomes mechanical. Remove serialization before tuning models. Keep LiveKit close to the user path. Tighten endpoint policy until clipping appears, then back off slightly. Let faster-whisper expose stable partials without waiting for decorative formatting. Force Gemini Flash to produce clause-shaped answers under length pressure. Begin Piper on the first safe clause. Start playback on the smallest stable audio reserve that avoids stutter. Treat interruption as a separate latency budget with its own thresholds and tests. None of these steps are glamorous. All of them are cumulative.

The trace also protects against false victories. A faster average STT time may be meaningless if p95 interruption recovery worsens. A shorter reasoning stage may simply be hiding increased TTS work caused by over-verbose first clauses. The system therefore needs percentile views and turn-class segmentation: clean turns, interrupted turns, resumed turns, failed turns. Real-time quality emerges from the tail as much as the median because users remember the awkward pauses and overlaps far more vividly than the smooth median exchange.

That is the right final picture for this project: not a mascot, not a chatbot persona, not an abstract model stack, but a timed room instrument. LiveKit carries sound. faster-whisper extracts committed text. Gemini Flash plans the reply under a strict clause budget. Piper makes speech from that plan. The system succeeds only when those stages behave as one measured pipeline and when interruption can cut across the pipe without leaving stale audio behind. Conversational quality here is simply cumulative latency made small, visible, and correctable.
